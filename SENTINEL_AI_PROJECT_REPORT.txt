1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Sentinel AI is a fully autonomous, self-healing distributed IoT monitoring
system built in Python. The system continuously collects real-time hardware
and network metrics from a device (targeted at Raspberry Pi / edge IoT nodes),
detects anomalies using multiple AI/statistical methods, diagnoses root causes
using LLMs (local and cloud), and executes automated recovery actions — all
without human intervention.

3.2  MONITORING AGENT  (agents/monitoring/monitoring_agent.py)
──────────────────────────────────────────────────────────────
Publishes a "health.metric" event to the event bus every 5 seconds.

Metrics collected:

  CPU:
    - cpu_percent (overall utilisation)
    - cpu_count, cpu_freq_current
    - load_avg_1min / 5min / 15min
    - top_process_name, top_process_cpu, top_process_pid

  Memory:
    - memory_percent, memory_total_mb, memory_available_mb, memory_used_mb
    - swap_percent, swap_used_mb
    - top_process_name, top_process_memory

  Disk:
    - disk_percent, disk_total_gb, disk_used_gb, disk_free_gb
    - disk_read_mb, disk_write_mb (cumulative IO counters)

  Network:
    - bytes_sent_mb, bytes_recv_mb, packets_sent/recv, errors_in/out
    - ping_latency_ms (average to 8.8.8.8 and 1.1.1.1)
    - packet_loss_percent
    - ping_success (bool)

  Power (simulated on macOS; replace with INA219/INA3221 on real hardware):
    - power_voltage_v  — input voltage, correlated with CPU load (± Gaussian noise)
    - power_current_a  — current draw (0.5A idle → 2.5A full load)
    - power_watts      — derived: voltage × current
    - power_quality    — 0–100 score; drops 8 pts per 1% voltage deviation
    - power_voltage_deviation_pct — absolute % deviation from nominal

  Power Simulation:
    - trigger_power_event(sag_volts, duration) injects a voltage sag
    - Resets automatically after duration_seconds via background thread

  MQTT / Sensors: disabled on macOS dev; stubs for real IoT hardware


3.3  ANOMALY DETECTION AGENT  (agents/anomaly/anomaly_detection_agent.py)
──────────────────────────────────────────────────────────────────────────
Fully adaptive — no hardcoded thresholds anywhere. All detection bounds
are computed at runtime from the live data stream.

AdaptiveMetricBaseline (per metric):
  - Long window: 300 readings (~25 min) for stable statistics
  - Short window: 20 readings (~100 s) for rate-of-change
  - Warmup: 30 readings required before any anomaly can fire
  - Anti-inflation: uses only the lower 60% (below upper mild fence) of data
    as the "normal" population — sustained spikes don't inflate the baseline

Detection Methods (all adaptive, no static numbers):

  1. IQR Outlier (Tukey Fence)
     - Mild fence:    Q3 + 1.5*IQR  → medium severity
     - Extreme fence: Q3 + 3.0*IQR  → high/critical severity
     - Both bounds are percentiles of observed data

  2. Adaptive Z-Score
     - z = |value - mean| / std
     - Base threshold: 2.5σ (configurable)
     - High-variance metrics (CV > 30%) get +0.5σ tolerance
     - Severity scales with z/threshold ratio

  3. Trend Elevation (time-series)
     - Fires when ALL of the last 5 readings exceed mean + 1.5*std
     - Catches gradual memory leaks or slow CPU creep

  4. Rate-of-Change Spike
     - Baseline: std of first-order differences over last 20 readings
     - Fires if step change > mean_change + 4.0 * change_std
     - Catches sudden vertical jumps (e.g. CPU 20% → 90% in one cycle)

  5. Isolation Forest (ML Layer 1, sklearn)
     - Multivariate point-in-time anomaly detection
     - Trained after 50+ readings; retrains every 15 min
     - contamination=0.1, n_estimators=100
     - Fires as anomaly type "ml_isolation_forest"

  6. Keras LSTM Autoencoder (ML Layer 2)
     - File: agents/anomaly/keras_lstm_detector.py
     - Features: [cpu_percent, memory_percent, disk_percent]
     - Window: 20 readings; LSTM Encoder (32 units) → RepeatVector → LSTM Decoder
     - Trains after 60+ sequences (~6.5 min of data); retrains every 500 readings
     - Anomaly threshold: err_mean + 3.0 * err_std (reconstruction MSE)
     - Uses PyTorch backend; PYTORCH_ENABLE_MPS_FALLBACK=1 for Apple Silicon
     - Fires as anomaly type "ml_lstm_autoencoder"

Deduplication Gates:
  - Persistence gate: metric must be anomalous for ≥2 consecutive readings
  - Cooldown gate: 5-minute suppression after an anomaly fires per metric

Excluded Metrics (too noisy or not meaningful for anomaly detection):
  disk.disk_read_mb, disk.disk_write_mb, network.bytes_*, network.packets_*,
  network.errors_*, mqtt.mqtt_latency_ms, cpu.cpu_count, cpu.cpu_freq_current,
  cpu.top_process_cpu, cpu.top_process_pid, cpu.load_avg_*,
  memory.top_process_memory, memory.memory_total_mb, memory.memory_available_mb,
  memory.memory_used_mb, memory.swap_used_mb, disk.disk_total_gb,
  disk.disk_used_gb, disk.disk_free_gb, sensors.sensor_count, sensors.sensors_online,
  power.power_quality, power.power_watts, power.power_voltage_deviation_pct


3.4  DIAGNOSIS AGENT  (agents/diagnosis/diagnosis_agent.py)
───────────────────────────────────────────────────────────
Triggered by "anomaly.detected" events. Runs diagnosis in a BACKGROUND THREAD
so slow LLM calls never block the event bus.

AI Stack (priority order):
  1. Groq  (llama-3.1-8b-instant, cloud, ~10× faster than Ollama)
     - Read from GROQ_API_KEY env var or .env file
     - Uses JSON response_format for structured output
  2. Ollama  (llama3.2:3b, local, no data leaves device)
     - Checked via ollama.list() at startup
     - System prompt + user prompt, format='json'
  3. OpenAI  (gpt-4o, disabled — no billing credits)
  4. AWS Bedrock  (claude-3-sonnet, disabled — requires certs)
  5. Rule-based  (instant fallback, always available)

Rule-based Diagnosis:
  - Loads rules from config/diagnosis_rules.yaml
  - Each rule: conditions (metric, operator, threshold), diagnosis template, actions
  - Templates support {metric.name} and bare {name} placeholder substitution
  - Process name, PID, etc. injected from anomaly string_context

LLM Prompt includes:
  - Metric name, detection type, current vs expected value, severity
  - 1-hour system context (mean/max/min per metric type from DB)
  - Exact list of available recovery action names
  - Instruction to respond as JSON only

Output: "diagnosis.complete" event with:
  - diagnosis_id, diagnosis, root_cause, confidence
  - recommended_actions (list of action names)
  - methods_used (e.g. ["rule_based", "groq_ai"])


3.5  RECOVERY AGENT  (agents/recovery/recovery_agent.py)
─────────────────────────────────────────────────────────
Triggered by "diagnosis.complete" events. Executes recovery actions.

Available Actions:
  kill_process      — Smart PID-based kill via InstabilityRunner; targets the
                      top CPU-consuming process (uses top_process_pid from metrics)
  restart_mqtt      — sudo systemctl restart mosquitto
  reconnect_sensor  — Reconnect disconnected IoT sensors
  failover          — Switch to backup MQTT broker
  clear_cache       — Clear /tmp/sentinel_cache and /var/cache/sentinel
  restart_service   — Restart sentinel-agent service

Safety:
  - Cooldown: 5-minute cooldown per action type (no action loops)
  - max_retries: 3 attempts with retry_delay=5s
  - auto_recovery flag (can be disabled in config)
  - Records each action to DB for learning agent

3.7  SECURITY AGENT  (agents/security/security_agent.py)
──────────────────────────────────────────────────────────
Runs every 30 seconds. Publishes "security.threat" events.

Real checks performed:
  - Open Port Scan: compares listening ports against allowlist
    {22, 80, 443, 1883, 5001, 8883}; unexpected ports → medium threat
  - Connection Flood: >200 concurrent connections → medium/high threat
  - Privileged Process: root processes not in safe list → low threat

Demo Mode (demo_mode: true in config):
  - 4% chance per scan of injecting a synthetic threat from these templates:
      * SSH brute-force attempt (high)
      * Unusual data egress (critical)
      * Port scan detected (medium)
      * Suspicious payload pattern / C2 beacon (critical)
      * Auth anomaly / new geo login (high)
  - Keeps last 50 threats in memory for dashboard display

Security toasts bypass the warmup gate — always shown immediately.

5. MACHINE LEARNING STACK
--------------------------------------------------------------------------------

Layer 1 — Isolation Forest (sklearn)
  - Multivariate, point-in-time detection
  - Feature vector: all numeric flat metrics at one timestep
  - Trains when 50+ readings accumulated
  - Retrains every 15 minutes in background
  - Score < 0 (in sklearn) → anomaly; severity mapped from score magnitude

Layer 2 — Keras LSTM Autoencoder
  - File: agents/anomaly/keras_lstm_detector.py
  - Architecture:
      Input (20 timesteps × 3 features)
      → LSTM Encoder (32 units, tanh)
      → RepeatVector(20)
      → LSTM Decoder (32 units, tanh, return_sequences=True)
      → TimeDistributed Dense (3 outputs)
      → Reconstruction (MSE loss)
  - Features: cpu_percent, memory_percent, disk_percent (normalized 0–1)
  - Trains after 60 complete sequences (79 readings ≈ 6.5 min)
  - Periodic retraining every 500 readings
  - Anomaly threshold: mean_reconstruction_error + 3.0 × std
  - Thread-safe: training in background thread, predictions read locked copy
  - Severity: <3σ=low, 3–5σ=medium, 5–8σ=high, >8σ=critical
  - Requires: PYTORCH_ENABLE_MPS_FALLBACK=1 for Apple Silicon Macs

Both ML layers respect the same 5-minute cooldown as statistical detectors.

6. SIMULATION LAB
--------------------------------------------------------------------------------

Purpose: trigger real anomalies to demonstrate the end-to-end pipeline.

  CPU Spike:
    - Starts cpu_stress subprocess via InstabilityRunner
    - Uses Python multiprocessing to saturate all cores
    - Detected by IQR outlier / z-score within ~30 s
    - Diagnosis: Groq/Ollama recommends kill_process
    - Recovery: kills the stress process by PID

  Memory Pressure:
    - Allocates memory in chunks until memory_percent rises
    - Detected by trend_elevation (gradual) or IQR (sudden)

  Disk Fill:
    - Creates ~200 MB temp file in /tmp
    - Note: may not spike disk_percent on large modern disks (expected)

  Power Sag:
    - POST /api/simulate/start/power_sag
    - Calls monitoring_agent.trigger_power_event(sag_volts=0.75, duration=60)
    - Drops voltage by 0.75 V below nominal for 60 seconds
    - Detected by z-score and IQR (power_voltage_v)
    - Anomaly type: statistical_zscore or iqr_outlier on power.power_voltage_v

  Stop All:
    - POST /api/simulate/stop
    - Kills all InstabilityRunner subprocesses
    - Resets _power_sim_sag to 0.0

7. KEY BUG FIXES AND DESIGN DECISIONS (CUMULATIVE)
--------------------------------------------------------------------------------

1. Toast Warmup Gate
   Problem: On startup, adaptive baselines have no data. Every first reading
            looks like a spike. → Flood of false-alarm toasts.
   Fix: Suppress anomaly toasts for the first 3 minutes of runtime.
        Security toasts are exempt (always shown).

2. Threshold Detection Key Mismatch
   Problem: Rule config used bare key "cpu_percent" but metrics dict uses
            "cpu.cpu_percent". Rules never matched.
   Fix: _format_diagnosis() and _match_rule() now try both the full dotted key
        AND the bare key after the last dot.

3. Adaptive Threshold Inflation
   Problem: During a long CPU stress test, the rolling baseline filled with
            high values, raising the "normal" band. The anomaly hid itself.
   Fix: AdaptiveMetricBaseline.stats() computes mean/std only on values
        BELOW the mild Tukey fence (lower 60% of distribution).

4. Incident Flood (Multiple Instances)
   Problem: Starting main.py without killing old processes caused multiple
            instances writing to the same SQLite DB → cascade of duplicate
            incidents on the dashboard.
   Fix: Always run pkill -f "python.*main.py" before starting.
        Documented in the one-line start command.

5. PID False Anomalies
   Problem: top_process_pid changes every reading (different process at top).
            The anomaly detector treated these integer jumps as anomalies.
   Fix: Added cpu.top_process_pid to excluded_metrics.

6. Load Average False Anomalies
   Problem: load_avg_* correlates strongly with cpu_percent, causing double-
            alerts for every CPU spike.
   Fix: Added all load_avg_* metrics to excluded_metrics.

7. Skipped = Failed (Learning Agent)
   Problem: If all recovery actions were skipped (cooldown), learning agent
            marked the incident as recovery_failed, skewing success rates.
   Fix: all-skipped outcome is treated as "recovery_attempted" not "failed".

8. Stat Panel Error Handling
   Problem: If /api/incidents returned an error, the JS stat panel crashed
            with an uncaught exception, breaking the whole dashboard.
   Fix: API endpoint returns empty array [] on any DB error (never 5xx).
        JS stat panel handles empty array gracefully.

9. Diagnosis Background Thread
   Problem: Ollama LLM calls block for 2–5 seconds. On a single-threaded
            event bus, this would delay every subsequent event.
   Fix: DiagnosisAgent.process_event() immediately snapshots event data,
        then spawns a daemon thread for the actual LLM call and publish.

10. Power Card Excluded from Wrong Detection
    Problem: power_quality, power_watts, power_voltage_deviation_pct were
             redundant derived metrics that would cause spurious anomalies
             alongside the primary power_voltage_v detection.
    Fix: Added those three to excluded_metrics; only power_voltage_v and
         power_current_a are anomaly-detected.
